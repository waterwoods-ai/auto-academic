# Design: General-Purpose program.md for Autonomous Experiments

## Summary

Adapt autoresearch's `program.md` mechanism into a general-purpose system for auto-academic. The experiment-planner agent generates a custom `program.md` per hypothesis using a fixed spine template with freeform sections. Each generated `program.md` drives a fully autonomous experiment loop (NEVER STOP) with LLM-judged keep/discard and tiered iteration (parameters first, escalate to code changes).

## Design Decisions

| Decision | Choice | Alternatives Considered |
|----------|--------|------------------------|
| Experiment scope | General-purpose (all 7 patterns) | ML-only |
| Keep/discard mechanism | LLM-judged (qualitative) | Single scalar metric, multi-metric with priority |
| Iteration target | Tiered: params first, code second | Params only, code+params from start |
| program.md origin | Generated by experiment-planner per hypothesis | Static template with slots, freeform generation |
| Template structure | Fixed spine + freeform sections | Pure template, pure generation |
| Stopping | NEVER STOP (human interrupts) | Built-in exit criteria, hybrid with convergence-judge |

## Architecture

### Generation Flow

```
experiment-planner agent
    |
    +-- reads: hypothesis, venue profile, experiment-patterns.md
    +-- reads: program-spine.md (template)
    |
    +-- generates per hypothesis:
            experiments/programs/{hypothesis_id}/program.md
            experiments/programs/{hypothesis_id}/main.py
```

Each hypothesis gets its own `program.md`, code directory, and git branch (`experiment/{hypothesis_id}`). Hypotheses can run in parallel on separate agent sessions.

The planner selects the experiment pattern (from `experiment-patterns.md`) during spec generation (existing behavior), then uses that same pattern to guide both the `program.md` freeform sections and the initial `main.py` code.

### The Fixed Spine

`program-spine.md` has 6 sections. Three are fixed, three have freeform zones.

| Section | Fixed / Freeform | Purpose |
|---------|-----------------|---------|
| 1. Setup | Mixed | Branch creation, results.tsv init, read in-scope files |
| 2. Scope | Freeform | What files to modify, what's read-only, tunable params |
| 3. Evaluation | Freeform | LLM judgment criteria for keep/discard |
| 4. Output Format | Mixed | Required fields + experiment-specific metrics |
| 5. Logging | Fixed | results.tsv format, git tracking |
| 6. Experiment Loop | Fixed | The autoresearch loop + tiered escalation |

### Fixed Parts (identical across all generated program.md files)

**Setup:**
- Create git branch `experiment/{hypothesis_id}`
- Initialize `results.tsv` with header row
- Read all in-scope files for context

**Logging — results.tsv format (tab-separated):**

```
commit	status	tier	description	evaluation_summary
```

- `commit`: short git hash (7 chars)
- `status`: `keep`, `discard`, or `crash`
- `tier`: `param` or `code`
- `description`: what this iteration tried
- `evaluation_summary`: brief LLM judgment rationale

**Experiment Loop:**

```
LOOP FOREVER:

1. Look at git state: current branch/commit
2. Modify code (respecting current tier constraints)
3. Git commit
4. Run experiment: python main.py > run.log 2>&1
5. Read results from run.log
6. Evaluate: apply LLM judgment criteria from Evaluation section
   - Also check: does this still test the original hypothesis? (drift check)
7. If BETTER: keep the commit, advance branch
8. If WORSE or DRIFT: git reset to before this iteration
9. Log to results.tsv (do NOT commit results.tsv)
10. Tier escalation check: if last 5 iterations were all discards
    at Tier 1 (param), escalate to Tier 2 (code)

Crash handling: if run crashes, read tail -n 50 run.log.
If fixable (typo, import): fix and retry (max 3 attempts).
If fundamental: log crash, move on to next idea.

Timeout: if a run exceeds 10 minutes, kill and treat as crash.

NEVER STOP: Do not pause to ask the human. Loop indefinitely
until manually interrupted.
```

### Freeform Parts (planner generates per hypothesis)

**Scope — defines the experiment sandbox:**
- Files the agent CAN modify (and what kind of changes per tier)
- Tier 1 tunable parameters (explicit list)
- Tier 2 code change boundaries
- Files the agent CANNOT modify (data, validated instruments, etc.)
- Available dependencies/libraries

**Evaluation — LLM judgment criteria:**
- What makes a run BETTER (specific to experiment type and hypothesis)
- What makes a run WORSE
- Hypothesis drift check criteria
- Edge cases and ambiguity guidance

**Output Format — experiment-specific metrics:**
- Fixed fields: `runtime_sec`, `status`
- Experiment-specific fields generated by planner (e.g., `p_value`, `effect_size`, `accuracy`, `modularity`, etc.)

### Tiered Escalation

```
TIER 1: Parameter Tuning (default)
  Agent modifies ONLY values marked as "tunable" in Scope.
  Examples: thresholds, sample sizes, learning rates, feature subsets,
  statistical test parameters, simulation iterations.

  Escalation trigger: 5 consecutive discards at Tier 1.
  (Planner can lower this to minimum 3, but not raise it.)

TIER 2: Code Changes
  Agent can modify experiment code itself — change algorithms,
  add preprocessing, restructure pipeline, swap methods.

  Constraint: changes must still test the SAME hypothesis.
  Hypothesis drift = automatic discard regardless of metrics.

  No further escalation. Tier 2 is the ceiling.
```

## Integration with SKILL.md Pipeline

### Revised Stage 2: EXPERIMENT

```
Step 1: Plan Experiments (updated)
  experiment-planner generates:
  - experiments/specs/{hypothesis_id}-experiment.md  (existing)
  - experiments/programs/{hypothesis_id}/program.md  (NEW)
  - experiments/programs/{hypothesis_id}/main.py     (NEW)

Step 2: Execute (changed — autonomous)
  For each hypothesis:
  - Hand program.md to a fresh agent session
  - Agent runs autonomously (NEVER STOP)
  - Human monitors results.tsv across running experiments
  - Human interrupts each when satisfied

Step 3: Collect Results (new)
  After human stops the loops:
  - Read final branch state for each hypothesis
  - Gather results.tsv files
  - Pass to experiment-evaluator for claim support assessment

Step 4: Evaluate & Converge (existing, unchanged)
  - experiment-evaluator judges claim support
  - convergence-judge decides: proceed to WRITE or re-plan
```

### Key Changes from Current Design

- Step 2 is no longer orchestrator-driven. The `program.md` agent is fully autonomous.
- The inner loop safety cap (10 iterations) is removed — replaced by human judgment.
- Experiment-evaluator and convergence-judge operate only after human stops the loops.
- Multiple hypotheses can run in parallel (separate sessions, separate branches).

### What Stays the Same

- Experiment-planner still designs the experiment
- Experiment-evaluator still judges claim support
- Convergence-judge still gates transition to WRITE
- Escalation-analyst still handles reviewer-triggered re-experiments in outer loop

## File Layout

### New and Modified Plugin Files

```
auto-academic/
  skills/auto-academic/
    templates/
      program-spine.md              NEW: fixed spine template
    references/
      experiment-patterns.md        UPDATED: add program.md examples per pattern

  agents/
    experiment-planner.md           UPDATED: generate program.md + initial code
```

`autoresearch/program.md` and `autoresearch/train.py` remain as upstream reference — unchanged.

### Runtime Output (per pipeline run, not checked in)

```
workspace/
  experiments/
    programs/
      {hypothesis_id}/
        program.md                  generated by planner
        main.py                     initial experiment code
        data/                       experiment data (if any)
        results.tsv                 accumulated by autonomous loop (untracked)
        run.log                     latest run output
```

Each hypothesis gets its own git branch (`experiment/{hypothesis_id}`) and directory.

## Concrete Example

For hypothesis "LLM usage correlates with reduced critical thinking scores" using survey analysis:

**Scope (generated):**
- CAN modify: `main.py` — survey analysis pipeline
- Tier 1 tunables: `significance_threshold`, `control_variables`, `regression_model_type`, `survey_instrument_weights`, `sample_size`
- Tier 2 code changes: preprocessing pipeline, statistical test choice, feature engineering, data splitting
- CANNOT modify: `data/responses.csv`, `instruments/`
- Dependencies: pandas, scipy, statsmodels, pingouin

**Evaluation (generated):**
- BETTER if: effect size (Cohen's d) larger while p < 0.05; model fit improved (adj R² higher, AIC/BIC lower); more interpretable; statistical assumptions better satisfied
- WORSE if: significance lost without effect size gain; overfitting (cross-validation dropped); added complexity without improvement; hypothesis drift

**Output Format (generated):**
```
---
p_value:        0.003200
effect_size:    0.42
adj_r_squared:  0.31
n_samples:      450
runtime_sec:    12.3
status:         success
```
